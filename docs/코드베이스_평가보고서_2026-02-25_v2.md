## 2. 백엔드/DB 설계

### 스키마 구조

- 10개 테이블, 전부 RLS 활성화
- `ON DELETE CASCADE` 체인으로 참조 무결성 유지
- UUID PK + `gen_random_uuid()` 사용 — 예측 불가능 ID
- `CHECK` 제약: `payment_status`, `zone`, `question_type`, `category`, `role`

### 강점

1. **RLS 전 테이블 적용**: 보호되지 않은 테이블 없음
2. **SECURITY DEFINER RPC**: `auth.users` 직접 조회 필요한 경우 안전하게 래핑
3. **`handle_new_user()` 트리거**: 가입 시 자동 프로필 생성
4. **SuperAdmin RPC**: `assert_superadmin()` 헬퍼로 일괄 권한 확인

### 문제점

| 심각도 | 문제 | 위치 |
|--------|------|------|
| **높음** | ~~`sa_list_projects`가 `pr.user_id`로 JOIN — 실제 컬럼은 `owner_id`~~ **수정 완료** | `009_superadmin_rpc.sql:48` |
| 중간 | `orders` INSERT 정책이 `WITH CHECK (true)` — 누구나 주문 위조 가능 | `001_user_profiles.sql:288-290` |

## 4. 비즈니스 로직 / 기능 완성도

### AHP 수학 엔진 (95/100)

- **거듭제곱법(Power Method)**: 100회 반복, 1e-8 수렴 임계값 — 적절
- **일관성 비율(CR)**: Saaty RI 테이블 기반, CR > 0.1 경고 표시
- **가중 기하평균 집계(AIJ)**: Aczel & Saaty(1983) 방법론 정확 구현
- **BestFit 추천**: 브루트포스 17값 × 모든 쌍 탐색 — 정확하나 n>10 시 성능 고려 필요
- **전역 우선순위**: `calculateGlobalPriorities`로 계층 트리 정확 계산

**잠재적 이슈**:
- `lambdaMax` 계산 시 `priorities[i]` = 0일 경우 0 나눗셈 위험 (극단적 에지 케이스)
- BestFit에 크기 제한 없음 (n=15: 1,785회 AHP 실행/호출)

## 12. 결론

AHP Basic은 **연구용 AHP 플랫폼으로서 핵심 기능이 충실히 구현된 프로젝트**입니다. 수학적 정확성이 높고, 컴포넌트 구조가 일관되며, 연구자·평가자 이중 역할 지원이 잘 설계되어 있습니다.

**가장 시급한 3가지 과제**:
1. **Direct Input 라우팅 버그 수정** (기능 치명 결함)
2. **anon RLS 정책 스코핑 강화** (보안 위험)
3. **에러 처리 + Toast 연동 강화** (사용자 경험)

이 세 가지만 해결해도 종합 점수가 **81 → 88점** 수준으로 상승할 것으로 예상됩니다.
